Accessibility UX State Mapping is an expert design and UX skill for Accessibility UX. The focal concern is State Mapping, but the real objective is helping an AI create experiences that remain clear, measurable, and resilient when users are confused, rushed, or recovering from failure.

## When To Use

- Use this when Accessibility UX State Mapping shapes whether users understand what to do, what happened, and how to recover.
- Use this when visual polish is not enough and the design needs stronger behavioral logic.
- Use this when engineering handoff, QA, and measurement must preserve the real design intent.
- Use this when the AI is creating user-facing systems that must hold up beyond mockups.

## Core Principles

- Users need decision clarity more than decorative confidence.
- Hierarchy, copy, and state behavior form one interface language.
- A feature is not designed until error, empty, and recovery paths are designed too.
- Measurement belongs in design because good intentions are not usable evidence.
- A handoff should preserve the meaning of the experience, not just its arrangement.

## Workflow

1. Start with the user decision, the task flow, and the highest-confusion moments.
2. Design the state model, the content model, and the hierarchy together.
3. Map empty, slow, error, and retry behavior before concluding the design is finished.
4. Create handoff specs that preserve interaction logic rather than only static arrangement.
5. Define the signals that would confirm the design works or fails after launch.
6. Capture the reasoning so later AI work can improve the experience without flattening it.

## Artifacts

- A decision and state map for Accessibility UX State Mapping.

## Tradeoffs

- Visual distinctiveness versus immediate usability.
- Flexible local design versus system-wide coherence.
- Faster shipping versus edge-state completeness.
- Dense information versus user confidence under stress.

## Signals To Watch

- User hesitation, abandonment, or reversal on the targeted flow.
- Support or research evidence showing misread hierarchy or unclear recovery.
- Design-to-implementation drift after handoff.
- Accessibility and state coverage defects after release.
- Post-launch changes that reveal the original design intent was under-specified.

## Review Checklist

- [ ] The key user decision inside Accessibility UX State Mapping is explicit.
- [ ] State behavior covers empty, slow, error, and recovery paths.
- [ ] Copy, hierarchy, and interaction reinforce the same meaning.
- [ ] Handoff materials are strong enough for engineering and QA.
- [ ] Measurement exists for validating the experience after launch.
- [ ] The design survives pressure without collapsing into superficial polish.

## Common Failure Modes

- Designing the hero path while leaving confusing states to implementation.
- Using refined visuals to mask weak behavioral logic.
- Treating critique as preference instead of structured reasoning.
- Handing off screens that do not explain how the experience behaves.
